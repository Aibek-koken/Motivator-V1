# Техническое описание проекта Motivator

## 1. Назначение

`Motivator` — это Telegram-бот для персональной мотивационной поддержки. По текущему коду он совмещает четыре основных сценария:

1. сохранение личных артефактов прошлого (`/memory`);
2. управление задачами (`/tasks`);
3. визуализацию желаемого будущего (`/future`);
4. диалоговую психологическую поддержку (`/psycho`).

Бот работает как stateful chat application поверх Telegram, хранит пользовательские данные в Supabase и использует LLM через Groq API для генерации части ответов.

Важно: в репозитории есть следы старого позиционирования и нейминга:

- в `README.md` проект называется `AnchorAI`;
- в `bot.py` и текстах интерфейса используется имя `Qaiyrat`;
- папка репозитория называется `Motivator`.

Это не ломает код, но означает, что продуктовая и техническая документация уже разошлись.

## 2. Технологический стек

- Язык: `Python`
- Telegram framework: `python-telegram-bot==21.6`
- LLM provider SDK: `groq==0.11.0`
- Database/API client: `supabase==2.7.4`
- Env management: `python-dotenv==1.0.1`
- Scheduler dependency: `APScheduler==3.10.4`

## 3. Точка входа и runtime-модель

Основная точка входа: [`bot.py`](./bot.py)

### Что делает `bot.py`

- загружает переменные окружения через `load_dotenv()`;
- собирает `Application` из `python-telegram-bot`;
- настраивает сетевые таймауты через `HTTPXRequest(connect_timeout=30, read_timeout=30)`;
- регистрирует bot commands:
  - `/memory`
  - `/tasks`
  - `/future`
  - `/psycho`
  - `/help`
- подключает `ConversationHandler` для основных режимов;
- запускает long polling через `app.run_polling(...)`.

### Runtime-характеристики

- Модель доставки обновлений: `long polling`, не webhook.
- Состояние диалогов хранится в двух местах:
  - временное состояние в `context.user_data`;
  - долговременное состояние в Supabase.
- Бот однопроцессный на уровне приложения и не содержит отдельного worker-слоя.

## 4. Архитектура по слоям

Проект фактически разбит на три слоя:

### 4.1. Transport / orchestration

Файл: [`bot.py`](./bot.py)

Задача слоя:

- принять апдейты от Telegram;
- отправить их в нужный handler;
- склеить ConversationHandler и callback-based UI.

### 4.2. Feature handlers

Папка: [`handlers/`](./handlers)

Каждый handler отвечает за свой пользовательский сценарий:

- `onboarding.py` — первичный `/start`-онбординг;
- `memory.py` — хранилище воспоминаний;
- `tasks.py` — задачи;
- `future.py` — vision board / визуализатор будущего;
- `psycho.py` — психологическая сессия;
- `daily.py` — заготовка ежедневного check-in;
- `sos.py` — legacy-обработчик кризисных сообщений и `/anchor`.

### 4.3. Service layer

Папка: [`services/`](./services)

- `db.py` — доступ к Supabase и CRUD-функции;
- `ai.py` — генерация AI-ответов и prompt orchestration.

## 5. Пользовательские сценарии

## 5.1. `/start` — онбординг

Файл: [`handlers/onboarding.py`](./handlers/onboarding.py)

Сценарий состоит из 4 шагов:

1. цель пользователя;
2. почему это важно;
3. пример прошлой победы;
4. образ/фото.

### Что сохраняется

- пользователь создается или обновляется в таблице `users`;
- цель и мотивация записываются в `future_profile`;
- первая победа записывается в `victories`.

### Техническое замечание

Обработчик ожидает, что запись в `future_profile` уже существует, и вызывает `update_future_profile(...)`, а не `init_future_profile(...)`. При чистой базе это может быть проблемой, если строки для пользователя еще нет.

## 5.2. `/memory` — архив прошлого

Файл: [`handlers/memory.py`](./handlers/memory.py)

Поддерживает:

- сохранение фото;
- сохранение ссылок;
- сохранение заметок/текста;
- просмотр по типам;
- закрепление;
- удаление.



### Типы записей

Поддерживаемые типы на уровне кода и БД:

- `photo`
- `text`
- `link`
- `note`
- `voice`

### UX-модель

- фото можно сохранить сразу или с подписью;
- ссылка сохраняется после запроса короткого описания;
- текст из `/memory <текст>` сохраняется как `note`;
- список строится через inline-кнопки и callback data вида `mem_*`.

### Дополнительная логика

Есть эвристика поиска "победы" по ключевым словам. Если запись похожа на достижение, она дополнительно копируется в `victories`.

## 5.3. `/tasks` — список задач

Файл: [`handlers/tasks.py`](./handlers/tasks.py)

Поддерживает:

- просмотр активных задач;
- добавление задачи через `/tasks <текст>`;
- добавление задачи кнопкой;
- завершение;
- удаление;
- просмотр выполненных.

### Особенности реализации

- приоритет определяется эвристически по ключевым словам;
- активные задачи показываются отдельными сообщениями;
- выполнение срочной задачи добавляет запись в `victories`.

### Хранилище

Используется таблица `tasks`.

## 5.4. `/future` — визуализатор будущего

Файл: [`handlers/future.py`](./handlers/future.py)

Сценарий реализует vision board:

- добавление элемента по категории;
- просмотр всей доски;
- AI-погружение по собранным элементам;
- удаление элементов.

### Категории элементов

- `machine`
- `home`
- `body`
- `income`
- `travel`
- `relationship`
- `skill`
- `quote`
- `other`

### AI-часть

Для `/future` используется только `generate_vision_immersion(...)`.
Уточняющие вопросы по категориям сейчас не генерируются моделью, а захардкожены в `handlers/future.py`.

### Хранилище

Используется таблица `vision_items`.

Важно: эта таблица используется в коде, но отсутствует в `schema.sql`.

## 5.5. `/psycho` — психолог-мотиватор

Файл: [`handlers/psycho.py`](./handlers/psycho.py)

Это самый сложный сценарий в проекте.

### Этапы

1. проверка пользователя;
2. проверка активной сессии;
3. onboarding психопрофиля, если он неполный;
4. сбор триггера текущего состояния;
5. сбор намерения от сессии;
6. запуск AI-сессии;
7. продолжение диалога с историей;
8. завершение с оценкой состояния.

### Психологический профиль

Собираются поля:

- `stress_pattern`
- `energy_killer`
- `support_type`
- `comeback_resource`

Эти данные используются для адаптации стиля AI-ответов.

### Сессионная модель

- в `users` хранится флаг `in_psycho_mode`;
- в `psycho_sessions` хранится история диалога, счетчик сообщений и оценки настроения;
- `context.user_data` используется как временный runtime-контекст.

### Fallback-поведение

Есть `psycho_handle_message(...)`, который подхватывает переписку вне ConversationHandler, если в БД стоит `in_psycho_mode = true`. Это защита от потери in-memory state, например после перезапуска бота.

## 6. Слой AI

Файл: [`services/ai.py`](./services/ai.py)

### Провайдер

Сейчас код использует `Groq`, а не Anthropic.

Модель:

- `llama-3.3-70b-versatile`

### Основные генераторы

- `generate_psycho_onboarding_response(...)`
- `generate_psycho_response(...)`
- `generate_vision_immersion(...)`
- `generate_sos_response(...)`
- `generate_daily_question(...)`
- `generate_anchor_response(...)`

### Prompt-подход

- в коде много системных промптов, ориентированных на прямой тон;
- есть общий блок `QUOTES_CONTEXT`, который подмешивает банк цитат и притч;
- AI-ответы ограничены по длине через `max_tokens`.

### Практический вывод

LLM здесь не управляет общей навигацией приложения. Навигация, состояния и UI-кнопки детерминированы кодом. Модель отвечает только за текстовую генерацию внутри уже заданного сценария.

## 7. Слой БД

Файл: [`services/db.py`](./services/db.py)

Подключение создается лениво через singleton-подобный `_client`.

Ожидаемые env-переменные:

- `SUPABASE_URL`
- `SUPABASE_KEY`

### Таблицы, которые реально использует код

- `users`
- `future_profile`
- `memories`
- `tasks`
- `victories`
- `psycho_sessions`
- `daily_checkins`
- `psycho_profile`
- `vision_items`

### Таблицы, описанные в `schema.sql`

В `schema.sql` сейчас есть:

- `users`
- `future_profile`
- `memories`
- `tasks`
- `victories`
- `psycho_sessions`
- `daily_checkins`

### Критичное расхождение

В `schema.sql` отсутствуют таблицы:

- `psycho_profile`
- `vision_items`

При этом код активно использует обе таблицы. Значит, SQL-схема не соответствует текущему приложению и не поднимает проект полностью в чистой Supabase-базе.

## 10. Несоответствия и технический долг

Ниже не гипотетические риски, а прямые расхождения, видимые в коде.

### 10.1. `README.md` устарел

В `README.md`:

- упоминается Anthropic, но код использует Groq;
- описан более узкий продукт, чем есть в текущем коде;
- перечислены не все активные функции.

### 10.2. `schema.sql` не покрывает текущее приложение

Отсутствуют:

- `psycho_profile`
- `vision_items`

Без них `/psycho` и `/future` в полном объеме не смогут работать на новой базе.

### 10.3. В проекте есть legacy-модули с битой связностью

`handlers/daily.py` и `handlers/sos.py` импортируют функции `get_profile` и `get_all_profiles`, которых в текущем `services/db.py` нет.

Это означает:

- эти модули написаны под старую версию data layer;
- в текущем состоянии они не готовы к безопасному подключению;
- сейчас они не участвуют в `bot.py`, что фактически скрывает проблему.

### 10.4. `APScheduler` установлен, но не встроен в runtime

Зависимость присутствует в `requirements.txt`, но в `bot.py` нет инициализации планировщика. Значит ежедневный check-in сейчас не запускается автоматически.

### 10.5. `/help` объявлен как bot command, но отдельный handler не зарегистрирован

В `post_init(...)` команда есть, но явного `CommandHandler("help", ...)` в `bot.py` нет.

### 10.6. Есть следы неполной миграции продуктовой модели

Например:

- `future_profile` используется и как профиль мечты, и как источник данных для AI;
- часть старых функций в `services/ai.py` сохранена "для совместимости";
- `handlers/sos.py` работает с полями `goal` и `why`, а основной код уже опирается на `dream` и `dream_why`.

## 11. Архитектурная оценка

Сильные стороны текущей реализации:

- понятное разбиение по feature handlers;
- простой и прозрачный transport layer;
- изолированный DB layer;
- LLM используется точечно, а не как скрытый orchestrator;
- большинство пользовательских сценариев можно расширять локально.

Слабые стороны:

- отсутствует единый актуальный контракт данных;
- схема БД отстает от кода;
- часть функциональности существует как legacy-слой и потенциально вводит в заблуждение;
- нет тестов;
- нет явной стратегии миграций схемы;
- нет разделения на dev/prod runtime и background jobs.

## 12. Что нужно, чтобы проект считался технически целостным

Минимальный набор работ:

1. синхронизировать `schema.sql` с фактическими таблицами `psycho_profile` и `vision_items`;
2. обновить `README.md` под реальный стек и реальные команды;
3. либо удалить `handlers/daily.py` и `handlers/sos.py`, либо довести их до совместимости с текущим `services/db.py`;
4. добавить обработчик `/help` или убрать команду из `set_my_commands`;
5. определить, нужен ли `APScheduler` в реальном runtime, и если нужен — встроить его в запуск;
6. унифицировать naming: `Motivator` / `AnchorAI` / `Qaiyrat`.

## 13. Краткий вывод

Текущее приложение — это не просто "мотивационный бот", а небольшой conversational system с четырьмя отдельными доменами: память, задачи, визуализация будущего и психологическая поддержка.

Главная техническая проблема проекта не в сложности кода, а в рассинхроне между:

- документацией,
- SQL-схемой,
- активным кодом,
- legacy-модулями.

Если описывать проект честно по текущему состоянию, то это Python Telegram bot на `python-telegram-bot` + `Supabase` + `Groq`, работающий через polling и ConversationHandler, с уже сформированной feature-архитектурой, но без завершенной консолидации данных и инфраструктуры.
